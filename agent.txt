1. ARCHITECTURE & IDENTITY MODEL
================================================================================

IDENTITY MODEL
- machineId: Permanent device identity, stored in localStorage (wisp_machine_id)
  + cookie fallback (wisp_mid, 400-day max-age). Created via crypto.randomUUID().
- sessionId: 30-minute visit window, stored in localStorage (wisp_session_state).
  Shared across tabs on same origin.

wisp_session_state format:
{
  "sessionId": "uuid",
  "startedAt": 1700000000000,
  "lastActivityAt": 1700000030000,
  "entryUrl": "https://mysite.com/"
}

Session closure is server-side via closeStaleSessions cron (every 10 min).

DATA MODEL (Convex schema)

machines table:
  machineId (string, PK), userId (string?), firstSeenAt (number),
  lastSeenAt (number), firstSeenDate (string "YYYY-MM-DD"),
  visitCount (number), userAgent (string?), platform (string?),
  referrer (string?), screen (string?), ip (string?), country (string?),
  region (string?), city (string?)

sessions table:
  sessionId (string, PK), machineId (string, FK), startedAt (number),
  lastActivityAt (number), endedAt (number?), durationMs (number?),
  eventCount (number), errorCount (number), isReturning (boolean),
  entryUrl (string?), exitUrl (string?)

events table:
  sessionId (string, FK), machineId (string, FK),
  type ("pageview" | "interaction" | "error" | "custom"), name (string),
  payload (any?), url (string), timestamp (number)

dailyStats table:
  date (string "YYYY-MM-DD"), newUsers (number), returningUsers (number),
  totalSessions (number), totalErrors (number), totalEvents (number),
  avgSessionDurationMs (number)

SAMPLING
Per-machine decision cached in localStorage under wisp_sampled_in.
  samplingRate: 1 → always track (default)
  samplingRate: 0 → track nobody
  samplingRate: 0.5 → track ~50% of distinct machines

GEO-IP RESOLUTION
Server-side in HTTP Action:
  1. IP: cf-connecting-ip → x-forwarded-for → x-real-ip
  2. Cloudflare geo: cf-ipcountry (all plans), cf-region (Business+),
     cf-ipcity (Business+)
  3. Fallback: ip-api.com/json/{ip}?fields=country,regionName,city
     (free, 45 req/min, no API key)

EVENT LIFECYCLE
  1. Browser event occurs (click, pageview, error, manual track())
  2. SDK queues in EventQueue (in-memory buffer)
  3. Queue flushes (timer/maxSize/pagehide) → ConvexTransport.send()
     POST https://{deployment}.convex.site/ingest
     { events: [...] }
  4. Convex HTTP Action parses body, resolves geo, calls
     recordBatchWithGeo(events, geo)
  5. For each event: upsertMachine → upsertSession → insert event
  6. Crons: closeStaleSessions (every 10min), computeDailyStats (00:15 UTC)

2. QUICKSTART
================================================================================

1. CREATE A CONVEX PROJECT
   npx convex dev
   Get a deployment URL: https://happy-animal-123.convex.cloud

2. COPY THE CONVEX FUNCTIONS
   Copy convex/ directory into your project:
   your-project/convex/
     schema.ts, events.ts, stats.ts, http.ts, dashboard.ts, crons.ts
     _generated/ (auto-generated by npx convex dev)
   Run: npx convex dev

3. INSTALL THE SDK
   npm install @renderdragonorg/wisp

   Initialize early:
   import { init } from "@renderdragonorg/wisp";
   init({ convexUrl: "https://happy-animal-123.convex.cloud" });

   Auto-tracks: machine ID, session, session_start event, pageviews,
   window.onerror/unhandledrejection, clicks on [data-wisp-id] elements.
   Batches every 5s or 20 events; flushes on tab close via sendBeacon.

4. OPEN THE DASHBOARD
   cd dashboard && npm install && npm run dev
   Set VITE_CONVEX_URL in dashboard/.env.local

5. VERIFY
   import wisp from "@renderdragonorg/wisp";
   wisp.track("test_event", { message: "hello!" });

3. CLIENT SDK
================================================================================

INSTALLATION
  npm install @renderdragonorg/wisp

INITIALIZATION
  import { init } from "@renderdragonorg/wisp";
  const client = init({ convexUrl: "https://happy-animal-123.convex.cloud" });
  Call once; repeated calls return existing singleton.

SINGLETON API (no reference needed after init)
  import wisp from "@renderdragonorg/wisp";
  wisp.track("event_name", { key: "value" });
  Methods throw "Not initialized" if called before init().

CONFIGURATION OPTIONS
  convexUrl         — Required (unless transport provided). Convex deployment URL.
  transport         — Custom transport, overrides convexUrl.
  sessionTimeoutMs  — 1800000 (30 min). Inactivity before new session.
  batchIntervalMs   — 5000 (5s). Queue auto-flush interval.
  batchMaxSize      — 20. Max buffered events before forced flush.
  autoTrackErrors   — true. Track window.onerror + unhandledrejection.
  autoTrackClicks   — true. Track clicks on [data-wisp-id] elements.
  autoTrackPageviews— true. Track SPA pageviews via pushState/replaceState.
  samplingRate      — 1. Fraction of machines to track (0–1).
  plugins           — []. Additional WispPlugin instances.
  redact            — (event) => event | null. Mutate/drop before queuing.
  debug             — false. Console-log internal SDK behavior.

TRACKING METHODS

  track(name, payload?)
    client.track("signup_completed", { plan: "pro" });
    wisp.track("signup_completed", { plan: "pro" });
    Event type inference:
      "pageview" → type pageview
      "click:*"  → type interaction
      everything else → type custom

  trackError(error, context?)
    Accepts Error object (captures message + stack) or plain string.
    wisp.trackError(new Error("Payment failed"), { component: "Checkout" });
    wisp.trackError("Something went wrong");

  identify(userId)
    Attach user identity (e.g. after login). Attaches userId to all subsequent
    events from this machine. Does not retroactively update past events.
    wisp.identify(data.session.user.id);

  reset()
    Clear user identity (e.g. after logout).
    wisp.reset();

  getMachineId() → string
    Returns stable machine ID.

  getSessionId() → string
    Returns current session ID.

  flush() → Promise<void>
    Force-flush event buffer.

  destroy()
    Tear down: uninstalls plugins, clears timer, emits final beacon flush.

AUTOMATIC TRACKING (3 built-in plugins, configurable via autoTrack* options)

  Error Plugin:
    Listens to window.onerror + window.onunhandledrejection.
    Calls trackError() with error + source metadata.

  Click Plugin:
    Event delegation on [data-wisp-id] elements.
    Extra data-wisp-* attributes collected into payload.
    <button data-wisp-id="signup_btn" data-wisp-plan="pro">Sign up</button>
    Produces: track("click:signup_btn", { plan: "pro" })

  Pageview Plugin:
    Fires pageview on init, patches history.pushState/replaceState,
    listens to popstate.

CUSTOM TRANSPORT
  import { init, type WispTransport } from "@renderdragonorg/wisp";

  const myTransport: WispTransport = {
    async send(events, opts) {
      await fetch("/api/analytics", {
        method: "POST",
        body: JSON.stringify(events),
        keepalive: opts.beacon,
      });
    },
  };
  init({ transport: myTransport });

CUSTOM PLUGIN
  import { init, type WispPlugin, type WispClientInternal } from "@renderdragonorg/wisp";

  const scrollPlugin: WispPlugin = {
    name: "scroll_depth",
    install(client: WispClientInternal) {
      window.addEventListener("scroll", () => {
        const depth = Math.round(
          (window.scrollY / (document.documentElement.scrollHeight - window.innerHeight)) * 100
        );
        if (depth >= 25 && depth % 25 === 0) {
          client.track("scroll_depth", { depth });
        }
      }, { passive: true });
    },
  };
  init({ convexUrl: "...", plugins: [scrollPlugin] });

EVENT QUEUE BEHAVIOR
  - Timer-based: every batchIntervalMs (default 5s)
  - Size-based: buffer reaches batchMaxSize (default 20)
  - Pagehide: visibilitychange → hidden / pagehide, beacon: true
    (sendBeacon if available, fetch + keepalive fallback)
  - Transport never throws — errors logged in debug mode, batch dropped

4. SUPABASE AUTH BINDING
================================================================================

OVERVIEW
  The Supabase integration links Supabase auth state to Wisp identity.
  When a user signs in/out via Supabase, Wisp automatically calls
  identify(reset()) so all analytics events carry the correct userId.

INSTALLATION
  Peer dependency (optional): @supabase/supabase-js ^2.110.0
  npm install @supabase/supabase-js
  (Only needed if using the Supabase auth binding)

IMPORT
  import { bindSupabase } from "@renderdragonorg/wisp/supabase";
  import { createClient } from "@supabase/supabase-js";

  const supabase = createClient("https://abc.supabase.co", "public-anon-key");
  bindSupabase(supabase);

HOW IT WORKS
  The bindSupabase function:
  1. Calls supabase.auth.getSession() immediately:
     - If a session exists with a user.id, calls wisp.identify(user.id)
     - This attaches the userId to all subsequent events
  2. Subscribes to onAuthStateChange:
     - SIGNED_IN → wisp.identify(session.user.id)
     - SIGNED_OUT → wisp.reset()
  3. Returns an unsubscribe function to clean up the listener

  Note: The initial session_start event may fire before identify() resolves
  (it's async). That event won't carry a userId. All subsequent events will.

USAGE WITH REACT / VUE / ETC.
  // After both Supabase and Wisp are initialized:
  import { bindSupabase } from "@renderdragonorg/wisp/supabase";
  import { createClient } from "@supabase/supabase-js";

  const supabase = createClient(
    import.meta.env.VITE_SUPABASE_URL,
    import.meta.env.VITE_SUPABASE_ANON_KEY
  );

  // In your app entry point or layout:
  bindSupabase(supabase);

UNSUBSCRIBE
  const unsubscribe = bindSupabase(supabase);
  // Later, e.g. on component unmount:
  unsubscribe();

EXAMPLE WITH INIT
  import { init } from "@renderdragonorg/wisp";
  import { bindSupabase } from "@renderdragonorg/wisp/supabase";
  import { createClient } from "@supabase/supabase-js";

  init({ convexUrl: "https://happy-animal-123.convex.cloud" });

  const supabase = createClient(
    "https://abc.supabase.co",
    "public-anon-key"
  );
  bindSupabase(supabase);

BUILD OUTPUT (after npm run build)
  dist/
    supabase.js   (ESM)
    supabase.cjs  (CommonJS)
    supabase.d.ts (TypeScript declarations)

TESTING (4 tests in tests/supabase.test.ts)
  - returns an unsubscribe function
  - calls getSession on init
  - calls onAuthStateChange on init
  - calls unsubscribe on returned function

DIAGRAM: Supabase Auth → Wisp Identity Flow
  User signs in via Supabase Auth
       │
       ▼
  onAuthStateChange("SIGNED_IN", session)
       │
       ▼
  wisp.identify(session.user.id)
       │
       ▼
  All subsequent tracked events carry userId

5. BACKEND — CONVEX FUNCTIONS
================================================================================

BACKEND FILES
  convex/schema.ts     — Table definitions, indexes, search indexes
  convex/events.ts     — recordBatch, recordBatchWithGeo, getSessionEvents,
                         getTopErrors
  convex/stats.ts      — closeStaleSessions, computeDailyStats, getDailyStats
  convex/http.ts       — HTTP Action at /ingest
  convex/dashboard.ts  — All dashboard queries
  convex/crons.ts      — Cron job definitions

SCHEMA (schema.ts) — Tables: machines, sessions, events, dailyStats
  Indexes: machines by machineId, sessions by sessionId/machineId,
  events by sessionId/machineId, dailyStats by date
  Search indexes: sessions entryUrl (for search), events name (for error
  search)

MUTATIONS

  events:recordBatch (public mutation)
    Primary ingestion. Accepts event array, processes oldest-first.
    For each event: upsertMachine → upsertSession → insert event row.
    Does NOT accept geo data.
    ctx.runMutation("events:recordBatch", { events: [...] });
    Returns { inserted: number }

  events:recordBatchWithGeo (internal mutation)
    Same as recordBatch but accepts geo object for IP/geo data.
    Called only by HTTP Action.
    ctx.runMutation(internal.events.recordBatchWithGeo, {
      events: [...],
      geo: { ip, country, region, city }
    });

QUERIES

  events:getSessionEvents(sessionId)
    Returns all events for a session, ascending by timestamp.

  events:getTopErrors(since)
    Returns most frequent error names in time range.
    [{ name: "TypeError", count: 42 }, ...]

HTTP ACTION (http.ts)
  POST https://{deployment}.convex.site/ingest
  SDK's ConvexTransport sends all events here.
  Steps: parse JSON → resolveGeo (IP + Cloudflare + ip-api.com fallback) →
  call recordBatchWithGeo → return 204 No Content with CORS headers.
  CORS: Access-Control-Allow-Origin: *

CRONS

  closeStaleSessions — Every 10 minutes
    Finds sessions with no activity > 30 min and endedAt not set.
    Sets endedAt + durationMs.

  computeDailyStats — 00:15 UTC
    Computes daily rollups for previous day:
    new machines, returning users, total sessions/errors/events,
    avg session duration. Uses upsert.

DASHBOARD QUERIES

  dashboard:getOverview(startDate, endDate)
    Returns totals + time series for chart area.
    Uses dailyStats if available; falls back to raw sessions.

  dashboard:listSessions(paginationOpts, sortBy, sortDir,
                         onlyReturning?, onlyWithErrors?, search?)
    Paginated, sortable, filterable session list.
    Search by entry URL (search index).

  dashboard:listEvents(paginationOpts, type?, search?)
    Paginated event list, filterable by type and name.

  dashboard:getErrorBreakdown(since)
    Top 8 error names by frequency.

  dashboard:getErrorDetails(errorName, since, limit?)
    Full event rows for a specific error name.

  dashboard:getMachineStats(machineId)
    Comprehensive single-machine stats: machine info, stats, top errors,
    top pages, last session. Returns null if not found.

  dashboard:getTopPages(startDate, endDate, limit?)
    Top pages by view count. [{ url, viewCount, uniqueMachines }, ...]

  dashboard:getPageVisitors(url, startDate, endDate)
    Machines that visited a specific page.

  dashboard:getPageViewsOverTime(startDate, endDate)
    Daily time series of page views. [{ date, pageViews, uniqueMachines }]

  dashboard:searchMachines(prefix)
    Autocomplete machine search by machineId prefix.

  dashboard:recomputeStats() (action)
    Manually runs daily stats computation for yesterday + today.
    Also calls closeStaleSessions first.

SETTINGS
  No env vars or secrets needed. ip-api.com works without API key.
  Modify resolveGeo in convex/http.ts for a different geo-IP provider.

6. DASHBOARD
================================================================================

SETUP
  cd dashboard && npm install
  Create dashboard/.env.local: VITE_CONVEX_URL=https://...
  npm run dev → http://localhost:5173

REAL-TIME UPDATES
  Uses Convex useQuery hooks (WebSocket subscriptions).
  Run npx convex dev in parent to regenerate types (dashboard symlinks
  convex/ → ../convex).

COMPONENTS

  StatCards(startDate, endDate)
    Top-level summary: new users, returning, sessions, pageviews, errors,
    avg session length.

  TrendChart(startDate, endDate)
    Side-by-side mini charts: Users (new+returning), Activity
    (sessions, events, errors). Chart-type toggle: Line/Bar/Area.

  DistributionChart(startDate, endDate)
    Pie/Donut/Radar charts of aggregated totals.

  SessionsTable(onSelectMachine)
    Paginated session list with search (300ms debounced), sort (start time,
    duration, event count, error count), filters (returning only, has errors).
    Click machine ID to open MachineView.

  ErrorBreakdownChart(sinceMs, onSelectError, selectedError)
    Horizontal bar chart of top 8 error names. Click bar for error details.

  ErrorDetails(errorName, sinceMs, onClose)
    Expands below ErrorBreakdownChart. Shows error name + occurrence count,
    each occurrence card (message, timestamp, URL, session ID, machine ID),
    console output in dark code block.

  TopPagesChart(startDate, endDate, onSelectPage, selectedPage)
    Horizontal bar chart of top 10 pages by view count. Click bar for
    machine visitors.

  PageVisitors(url, startDate, endDate, onSelectMachine, onClose)
    Expands below TopPagesChart. Table of machines that visited: machine ID,
    visit count, country, platform, first/last visit timestamps.

  MachineView(machineId, onClose)
    Full-page detail view: timestamps, stats, device/location info,
    top errors, top pages, last session. Accessed via session click or
    top-bar autocomplete search.

LAYOUT
  1. Header bar — title, machine search, date range, refresh button
  2. Stat cards — summary metrics
  3. Trend chart — two side-by-side mini charts
  4. Distribution chart — pie/donut/radar
  5. Error breakdown — bar chart + expandable error details
  6. Top pages — bar chart + expandable visitor table
  7. Sessions table — paginated session list
  Selecting a machine replaces entire dashboard with MachineView.

BUILD FOR PRODUCTION
  cd dashboard && npm run build
  Output: dashboard/dist/
  Deploy to any static host (Vercel, Cloudflare Pages, Netlify, S3, etc.)
  Fully static SPA; all data comes from Convex queries over WebSocket.

8. TESTING
================================================================================

RUNNING TESTS
  Full suite (108 tests): npm test
  Unit only:              npx vitest run tests/*.test.ts --exclude tests/convex-integration.test.ts
  Integration:            npx vitest run tests/convex-integration.test.ts
  Watch:                  npm run test:watch
  Single file:            npx vitest run tests/transport.test.ts

TEST STRUCTURE
  tests/
    setup.ts                    — jsdom config, stubs
    convex-integration.test.ts  — 27 tests against live Convex dev deployment
    index.test.ts               — 22 tests: init, track, trackError, identify, redact, sampling, custom transport
    transport.test.ts           — 12 tests: fetch, sendBeacon, fallback, keepalive, URL normalization
    plugins.test.ts             — 11 tests: error, click, pageview plugins
    queue.test.ts               — 7 tests: buffering, flushing, destroy, error tolerance
    machineId.test.ts           — 7 tests: localStorage, cookie, SSR fallback, error recovery
    session.test.ts             — 9 tests: creation, reuse, timeout, persistence
    supabase.test.ts            — 4 tests: bindSupabase, auth state change
    comprehensive.test.ts       — 6 tests: full lifecycle, Supabase auth, data integrity, sampling, transport failure

COMPREHENSIVE TEST COVERAGE
  - Full lifecycle: init → 4 event types → click plugin → error plugin →
    pushState/replaceState → identify/reset → flush → data integrity → destroy
  - Supabase e2e: pre-existing session → getSession resolves → userId on events →
    SIGNED_IN → SIGNED_OUT → unsubscribe
  - Supabase pre-existing user: session_start has no userId → later events carry it
  - Supabase no initial session: no userId before SIGNED_IN → callback sets it
  - Data integrity: per-type payload verification, consistent IDs, monotonic timestamps
  - Sampling + transport failure: samplingRate: 0 drops everything; rejection doesn't throw

INTEGRATION TESTS
  Run against live Convex dev deployment. Use unique machineId/sessionId per test.
  Helper functions callMutation() and callQuery() via fetch to Convex HTTP API.

UNIT TEST CONVENTIONS
  - vi.resetModules() in beforeEach for module-level singletons
  - Mock fetch with vi.spyOn(globalThis, "fetch")
  - vi.useFakeTimers() / vi.useRealTimers() for timer tests
  - jsdom provides window, document, navigator, location, history
  - Customize navigator with Object.defineProperty(globalThis.navigator, ...)

CI (GitHub Actions example)

  jobs:

      runs-on: ubuntu-latest

        - uses: actions/checkout@v4
